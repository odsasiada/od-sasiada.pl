# Story 3.6: Filtr katalogu po kategorii (server-side, tenant-scoped)

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

Jako **klient**,
chcę **filtrować katalog `/[tenant]` po kategorii (`?kategoria=<slug>`), wyliczany serwerowo**,
aby **szybciej znaleźć produkty** (decyzja D4; ryzyko R-S3.5 — przeciek cross-tenant przez filtr).

## Acceptance Criteria

1. **Filtr server-side w `getCatalog`** — `/[tenant]?kategoria=<slug>` zawęża listę do produktów tenanta w danej kategorii. `where` zawsze zawiera `tenant` ORAZ `_status: 'published'` PLUS dopasowanie kategorii w relacji `hasMany`.
2. **Produkt w wielu kategoriach pojawia się w KAŻDEJ** ze swoich kategorii (D4) — wynika z dopasowania „relacja zawiera kategorię".
3. **Bez parametru `?kategoria=` → wszystkie** opublikowane produkty tenanta (zachowane obecne zachowanie katalogu).
4. **Lista kategorii w UI tylko z tego tenanta** (`where { tenant }`) — linki/filtry pokazują wyłącznie kategorie dostawcy.
5. **Izolacja (R-S3.5):** nieznany/cudzy slug → wynik **pusty/zignorowany**, **NIGDY** cross-tenant. Rozwiązanie slug→kategoria jest tenant-scoped (`where { slug, tenant }`).
6. **Bez regresji** — render zdjęć (S3.3), koszyk, paginacja działają; `pnpm lint`/`pnpm test`/`pnpm dev` zielone.

## Tasks / Subtasks

- [ ] **Task 1 — Lista kategorii tenanta (UI)** (AC: 4, 5)
  - [ ] `src/lib/shop.ts`: `getCategories(tenantId): Promise<{ id; name; slug }[]>` — `payload.find({ collection: 'categories', overrideAccess: true, where: { tenant: { equals: tenantId } } })`. Mapuj minimalnie (name + slug).
- [ ] **Task 2 — Filtr w `getCatalog`** (AC: 1, 2, 3, 5)
  - [ ] Rozszerz `getCatalog(tenantId: number, categorySlug?: string)`. Bez `categorySlug` → obecne zachowanie (wszystkie published tenanta).
  - [ ] Z `categorySlug`: najpierw rozwiąż kategorię **tenant-scoped** — `payload.find({ collection: 'categories', where: { and: [{ slug: { equals: categorySlug } }, { tenant: { equals: tenantId } }] }, limit: 1 })`. Brak dopasowania → zwróć `[]` (pusto; Q1).
  - [ ] Dopasowanie produktów: dołóż do `where` produktów warunek relacji `hasMany`, np. `{ categories: { in: [categoryId] } }` (zweryfikuj operator dla relacji hasMany — `in`/`contains` — Q2). `where` MUSI nadal mieć `tenant` + `_status: 'published'` (R-S3.5).
  - [ ] Zachowaj dotychczasowe wzbogacanie o `image` (S3.3) i warianty.
- [ ] **Task 3 — UI filtra na stronie katalogu** (AC: 1, 3, 4)
  - [ ] `src/app/(frontend)/[tenant]/page.tsx` (server): odczytaj `searchParams.kategoria` (Next 16 — `searchParams` jest Promise, `await`). Pobierz `getCategories(tenant.id)` + `getCatalog(tenant.id, kategoria)`.
  - [ ] Renderuj nawigację filtra: linki `?kategoria=<slug>` + „Wszystkie" (czyści filtr). Server-rendered linki (bez stanu klienckiego) — wybór = nawigacja + re-render. Zaznacz aktywną kategorię.
- [ ] **Task 4 — Test integracyjny** (AC: 1, 2, 5) — *wzór: `tests/integration/*.integration.test.ts`*
  - [ ] Produkt A1 w kategoriach K1,K2 (tenant A); produkt A2 w K2. Filtr `K1` → tylko A1; `K2` → A1 i A2 (wielokategoryjny w każdej — D4).
  - [ ] Bez slugu → wszystkie published A.
  - [ ] **Cross-tenant:** slug kategorii tenanta B podany dla tenanta A → **pusto** (nie produkty B, nie produkty A); nieznany slug → pusto. Nigdy cross-tenant (R-S3.5).
- [ ] **Task 5 — Weryfikacja końcowa** (AC: 6)
  - [ ] `pnpm test` zielone, zero regresji (`orders-isolation`, `categories-isolation`); `pnpm lint:fix` + `pnpm lint` czyste; `fallow audit --format json --quiet` bez nowych blokad.
  - [ ] `pnpm dev`: filtr działa, produkt wielokategoryjny widoczny w każdej kategorii, zdjęcia (S3.3) i koszyk działają.

## Dev Notes

### Co to za historia (zakres ścisły)
S3.6 = **server-side filtrowanie katalogu po kategorii** + UI filtra. Domyka EPIC-3. **Poza zakresem:** tworzenie kolekcji/relacji (S3.4), izolacja CRUD panelu (S3.5), render zdjęć (S3.3 — tu tylko nie zepsuć).

### R-S3.5 (KLUCZOWE) — `tenant` ZAWSZE w `where`
`getCatalog` czyta przez `overrideAccess: true` (plugin nie pilnuje publicznych zapytań — izolacja jest RĘCZNA). Filtr kategorii to nowy wektor przecieku: gdyby dopasowanie kategorii pominęło `tenant`, klient mógłby zobaczyć produkty innego dostawcy. **Dwa miejsca muszą być tenant-scoped:** (a) rozwiązanie `slug→categoryId` (`where { slug, tenant }`), (b) zapytanie produktów (`where { tenant, _status, categories }`). Wzór: istniejący `getCatalog` (`shop.ts:38`) już trzyma `tenant`+`_status` — tylko dołóż warunek kategorii, nie luzuj reszty.

### Dlaczego rozwiązywać slug w obrębie tenanta
Sluggi kategorii nie są globalnie unikalne (S3.4: unikalność per-tenant nie wymuszona). Gdyby filtrować produkty wprost po slug bez rozwiązania w obrębie tenanta, slug tenanta B mógłby przypadkiem trafić. Rozwiązanie `slug→id` z `where { slug, tenant }` gwarantuje, że bierzemy kategorię TEGO dostawcy; nieznany/cudzy slug nie rozwiąże się → pusto (R-S3.5).

### Zapytanie relacji `hasMany`
Produkt↔kategoria to `relationship hasMany` (S3.4). Filtrowanie „produkty, których `categories` zawiera X" w Payload zwykle: `where: { categories: { in: [categoryId] } }`. **Zweryfikuj operator** dla relacji hasMany w zainstalowanej wersji (`in` vs `contains`) — Q2. Produkt z wieloma kategoriami naturalnie spełnia warunek dla każdej z nich (D4).

### Next 16 — `searchParams` async
W Next 16 (App Router, ten projekt) `params` i `searchParams` są **Promise** — `await` je w server component (jak już robione z `params` w `page.tsx`). `?kategoria=` → string | undefined.

### Render server-side, bez stanu klienckiego
Filtr to linki nawigacyjne (`<Link href="?kategoria=slug">`), nie client state — wybór kategorii = nawigacja + serwerowy re-render `getCatalog`. Nie potrzeba 'use client'; `Catalog.tsx` (client) dostaje już przefiltrowaną listę propsami. Granica hydratacji bez zmian.

- **`src/app/(frontend)/[tenant]/page.tsx`**: dodaj odczyt `searchParams`, listę kategorii, UI filtra. Zachowaj sekcję intro/`Catalog`.

### Czego NIE robić (anti-patterns)
- ❌ NIGDY nie pomijaj `tenant` w `where` filtra ani w rozwiązaniu slug (R-S3.5 — przeciek cross-tenant).
- ❌ Nie filtruj produktów wprost po slug bez rozwiązania `slug→id` tenant-scoped (slug nie jest globalnie unikalny).
- ❌ Nie rób filtrowania po stronie klienta (server-side w `getCatalog` — AC1).
- ❌ Nie gub `_status: 'published'` ani wzbogacania `image` (S3.3) przy dokładaniu filtra.
- ❌ Nieznany/cudzy slug nie może „po cichu" pokazać wszystkich cudzych — pusto/ignorowane, nigdy cross-tenant.

## Open Questions (do potwierdzenia przed/po dev)
- **Q1 — Nieznany/cudzy slug: pusto czy ignorować filtr.** AC dopuszcza oba („pusto/ignorowana"). Rekomendacja: **pusto** (jawne „brak produktów w tej kategorii") — bezpieczniejsze i mniej mylące niż ciche pokazanie wszystkich. Potwierdź.
- **Q2 — Operator zapytania relacji `hasMany`.** Rekomendacja: `where { categories: { in: [categoryId] } }`. Zweryfikuj w `payload@^3.85.1`, czy `in`/`contains` poprawnie dopasowuje relację hasMany (produkt z wieloma kategoriami w każdej z nich — D4).
- **Q3 — UI filtra.** Rekomendacja: pasek linków kategorii + „Wszystkie", aktywna podświetlona; bez stanu klienckiego (nawigacja query-param). Potwierdź zakres UI (czy ma być licznik produktów per kategoria — raczej nie w MVP).
