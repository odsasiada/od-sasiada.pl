# S2.1 — Kolekcja `DeliverySlots` (multi-tenant): okna + cutoff + capacity

Status: ☐ TODO · ready-for-dev · EPIC-2 · utworzono 2026-06-21

## Historyjka

Jako **dostawca** chcę **skonfigurować tygodniowe okna dostawy + cutoff + limit miejsc w dedykowanej kolekcji `DeliverySlots`**, aby **klient wybierał tylko realne terminy** (decyzje O1–O4).

## Kryteria akceptacji (z sprint-2.md)

- Nowa kolekcja `DeliverySlots` wpięta w `multiTenantPlugin.collections` (izolacja tenant gratis — dostawca widzi/edytuje tylko swoje).
- Pola wg SPIKE-S2: `weekday` (0–6, etykiety PL), `windowStart`/`windowEnd` (`HH:mm`), `cutoffDaysBefore` (int ≥ 0), `cutoffTime` (`HH:mm`), `capacity` (int ≥ 0); relacja `tenant` przez plugin.
- Pola/etykiety po polsku; walidacja sensowności: `windowEnd > windowStart`, `capacity ≥ 0`, format `HH:mm`.
- Brak slotów u tenanta = feature wyłączony (O8).

## Zależności

- SPIKE-S2 (kształt kolekcji + kontrakt `computeAvailableSlots` domknięte).

## Uwagi implementacyjne

- Kształt pól i uzasadnienia: patrz `SPIKE-S2.md` → „Wynik spike'a (a)". Reprezentacja: dzień = `number`, godziny = string `HH:mm`, cutoff = `(cutoffDaysBefore, cutoffTime)`.
- `reservedCount`/obłożenie **nie jest polem** — liczone z `orders` (S2.7). `DeliverySlots` trzyma tylko config.
- Wpięcie w multi-tenant analogicznie do `media`/`products` w `payload.config.ts`. Po dodaniu kolekcji: `pnpm payload:types` + `pnpm test` (live Docker Postgres) bez regresji.
- Ryzyko R-S2.4 (model/zakres): O3 = dedykowana kolekcja — domknięte przez SPIKE-S2, schemat nie powinien się zmieniać w trakcie.
