# Story 4.4: Naprawa deploya produkcji — crash `sharp`/libvips + build typecheck

Status: review

## Story

Jako **operator**,
chcę **żeby aplikacja faktycznie uruchamiała się na produkcji (Vercel) bez błędu 500**,
aby **móc dokończyć weryfikację end-to-end (S4.3) i przejść na pilota z pierwszym dostawcą**.

## Kontekst (co się dzieje)

Na produkcji **każda** trasa zwraca `500` z generycznym ekranem Payload „A server error
occurred. Reload to try again." — nie tylko `/admin`, ale też `/swieze-z-kaszub`,
`/api/tenants`, a nawet `/favicon.ico`. Logi runtime Vercel:

```
Error: Could not load the "sharp" module using the linux-x64 runtime
ERR_DLOPEN_FAILED: libvips-cpp.so.8.18.3: cannot open shared object file: No such file or directory
```

**Przyczyna źródłowa:** `sharp` jest importowany w `src/payload.config.ts`, więc moduł crashuje
przy starcie → globalne 500. W drzewie zależności były **dwie** wersje sharp:
- `sharp@0.34.5` — opcjonalna zależność **Next.js 16.2.9** (dla `next/image`),
- `sharp@0.35.2` — bezpośrednia zależność appki (przekazywana do Payload `buildConfig`).

Binarka `@img/sharp-linux-x64@0.35.2` linkuje `libvips-cpp.so.8.18.3`
(`@img/sharp-libvips-linux-x64@1.3.1`), ale przez symlinkowy układ node_modules pnpm Next.js
nie pakuje tego `.so` do funkcji serverless → `DLOPEN_FAILED`.

**Drugi, niezależny problem:** build prod nie kompilował się przez typecheck
`src/seed-production.ts:192` (`optionId: string | number`). Naprawione już w HEAD `2df6b5f`
(`Number(opt.<...>.id)`).

## Dev Notes

### Decyzje
- **Downgrade, nie hoist:** zamiast walczyć z tracingiem pnpm (`.npmrc node-linker=hoisted`),
  zrównujemy sharp do wersji którą Next.js i tak instaluje (`0.34.5`). Jedna wersja sharp =
  jeden libvips = brak niejednoznaczności przy tracingu .so. Payload core wymaga luźno `0.32.6`,
  więc `0.34.5` go spełnia.
- `supportedArchitectures: {os:[linux], cpu:[x64]}` w package.json zostaje — gwarantuje, że
  linux-x64 binarka trafia do lockfile także przy instalacji na macOS.
- Cast `sharp as Parameters<typeof buildConfig>[0]['sharp']` w payload.config zostaje
  (Payload typuje pod sharp 0.32, runtime kompatybilny).

### Deploy
- Production leci **ręcznie** `vercel --prod` z brancha `feature/S4.1-seed-fixtures-local`
  (nie auto z `main`).

### Czego NIE robić
- ❌ Nie podbijać sharp Next.js w górę (brak kontroli nad peer-optional Next).
- ❌ Nie usuwać `supportedArchitectures` ani optionalDependency `@img/sharp-linux-x64`.

## Findings — handoff do S4.3 (NOWY bloker, nie sharp)

Po naprawie sharp `/admin` daje 200, ale `/swieze-z-kaszub` i `/api/tenants` nadal 500 z:

```
error: relation "tenants" does not exist
Failed query: select count(*) from "tenants" where "tenants"."slug" = $1
```

**Produkcyjna baza Supabase nie ma schematu Payload** — tabele nigdy nie zostały utworzone.
`/admin` (logowanie) nie odpytuje danych, więc działa; każdy data-route pada.

`@payloadcms/db-postgres` w **produkcji nie robi `push`** schematu (tylko dev). Potrzebne jedno z:
- uruchomienie **migracji** Payload na prod (`payload migrate`) — wymaga plików migracji, których
  projekt prawdopodobnie nie ma (dev jechał na `push`), albo
- jednorazowy `pnpm seed:production` z `DATABASE_URL` wskazującym Supabase (z `push` utworzy
  schemat + dane).
