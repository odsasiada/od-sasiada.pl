# Story 2.2: Wybór slotu dostawy w checkoucie (lista dostępnych)

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

Jako **klient**,
chcę **wybrać w checkoucie termin dostawy z listy dostępnych slotów (wyliczonej serwerowo)**,
aby **wiedzieć, kiedy dojedzie zamówienie** (decyzje O1–O8, EPIC-2).

## Acceptance Criteria

1. **Lista slotów liczona SERWEROWO.** Strona koszyka (`src/app/(frontend)/[tenant]/koszyk/page.tsx`, server component) wylicza dostępne wystąpienia slotów przez **istniejącą** czystą funkcję `computeAvailableSlots` z `src/lib/delivery-slots.ts` (NIE pisać jej od nowa) i przekazuje wynik do `CartView` **propsami**. Klient nie liczy slotów po swojej stronie.
2. **Źródła danych (odczyt frontendowy, NFR1 — izolacja tenant):** nowy *plain server util* (NIE `'use server'`) czyta z `overrideAccess: true` + **ręcznym `where { tenant }`**:
   - kolekcję `delivery-slots` (S2.1) → mapuje na `DeliverySlot[]`,
   - kolekcję `delivery-date-exceptions` (S2.8) → mapuje na `DateException[]`.
   `now` = `new Date()` wstrzyknięte do `computeAvailableSlots`.
3. **`reservedCount` = `0` (świadomy placeholder w tej historii).** Zamówienia NIE mają jeszcze relacji `deliverySlot` (dodaje ją S2.4), więc recount aktywnych zamówień na slot jest niemożliwy. Przekaż `reservedCount: 0` dla każdego slotu. Realny recount z `orders` + wymuszanie capacity = **S2.7** (poza zakresem). Skutek w tej historii: filtr capacity wyklucza tylko sloty z `capacity <= 0`.
4. **Sekcja widoczna tylko, gdy tenant ma skonfigurowane okna (O8).** Util zwraca `deliveryEnabled` = (liczba rekordów `delivery-slots` tego tenanta > 0). Gdy `deliveryEnabled === false` → sekcja wyboru slotu **ukryta**, checkout działa jak dziś (bez slotu w payloadzie). Gdy `deliveryEnabled === true` → sekcja widoczna, wybór **wymagany** (O8).
5. **Tylko realne sloty na liście.** Sloty zamknięte (po cutoffie), przeszłe/rozpoczęte, z wyłączonych dat (O7) oraz z `capacity <= 0` są **niewidoczne** — gwarantuje to `computeAvailableSlots` (już przetestowane, 15 testów jednostkowych). Lista posortowana po dacie i godzinie (gwarantuje funkcja).
6. **Wybór jednego slotu i przekazanie do payloadu checkoutu.** Klient wybiera dokładnie jeden dostępny slot; wybór (identyfikator slotu **oraz** konkretna data wystąpienia `YYYY-MM-DD`) jest przekazywany do akcji `placeOrder`. W tej historii wartość jest **tylko przenoszona** w sygnaturze akcji (parametr opcjonalny). **Autorytatywna walidacja cutoffu = S2.3, capacity = S2.7, snapshot w zamówieniu = S2.4** — tu NIE walidujemy serwerowo i NIE zapisujemy do `orders`.
7. **Wymuszenie wyboru po stronie klienta (gdy `deliveryEnabled`).** Przy `deliveryEnabled === true` przycisk „Złóż zamówienie" jest zablokowany dopóki klient nie wybierze slotu. Gdy `deliveryEnabled === true`, ale lista dostępnych slotów jest pusta → komunikat PL „Brak dostępnych terminów dostawy — skontaktuj się z dostawcą." i zablokowany submit. (To client-side UX; twardy guard serwerowy to S2.3.)
8. **Granica hydratacji nienaruszona (NFR6, R-S2.5).** `CartView` nadal hydratuje się poprawnie; żaden nowy import nie wciąga `next/headers` do modułu czystego ani do komponentu klienckiego. `pnpm dev` startuje, checkout działa end-to-end (z oknami i bez okien), `pnpm lint` + `pnpm test` zielone bez regresji (w tym testy izolacji `orders-isolation`, `order-detail-idor`).

## Dev Notes

### Co to za historia (zakres ścisły)
S2.2 to **wyłącznie ścieżka odczytu + UI wyboru**: serwerowy odczyt slotów/wyjątków → `computeAvailableSlots` → propsy → picker w `CartView` → przeniesienie wyboru do akcji. **Poza zakresem (świadomie, kolejne historie):**
- **S2.3** — serwerowa walidacja cutoffu w `placeOrder` (klientowi nie ufamy).
- **S2.7** — recount z `orders` + capacity odporne na wyścig (transakcja, `SELECT … FOR UPDATE`).
- **S2.4** — snapshot slotu w zamówieniu (relacja `deliverySlot` na `orders` + widok w panelu/froncie/mailu).
- **S2.6** — PL-izacja treści maili.

Nie zapisujemy slotu do `orders` (brak pola) i nie re-walidujemy go serwerowo w tej historii — wartość jest tylko **przenoszona**, by S2.3/S2.4 miały na czym budować.

### Fundament JUŻ ISTNIEJE — nie wymyślać od nowa
- **`computeAvailableSlots(slots, exceptions, now, horizonDays?)`** w `src/lib/delivery-slots.ts:150` — czysta, deterministyczna, przetestowana (DST, cutoff, O7, O4, O8). To **jedyne źródło prawdy** „które sloty są dostępne". Wstrzykujesz `now`; reszta filtrowania (przeszłe/cutoff/wyłączone/pełne/zdegenerowane) dzieje się w środku. **NIE duplikować tej logiki w UI ani w utilu odczytu.**
- Typy do importu z tego samego modułu: `DeliverySlot`, `DateException`, `AvailableSlot`, `Weekday`.

### Granica hydratacji Turbopack (R-S2.5 / NFR6) — KRYTYCZNE
Architektura warstw jest celowa i powtarza sprawdzony wzór z koszyka/adresów:
- **Moduł czysty** (`delivery-slots.ts`) — bez `'use server'`, `next/headers`, `@/lib/auth`, Reacta, I/O. Już taki jest; **nie dotykać tej zasady.**
- **Plain server util** (`delivery-slots-read.ts`, NOWY) — importuje `payload`/`config`, ale **bez `'use server'`**. Wzór 1:1: `src/lib/cart.ts` (`getCartSnapshot`) i `src/lib/addresses.ts` (`getCustomerAddresses`). Importowany **tylko** z server-componentu (`koszyk/page.tsx`), NIGDY z `CartView` (client).
- **Plik akcji** (`actions.ts`, `'use server'`) — importuje `next/headers` **bezpośrednio**, typy parametrów **inline**. Nie importować tego pliku jednocześnie z server- i client-componentu. (To już respektowane — zachować przy dodaniu parametru.)
- **Client component** (`CartView`) — dostaje `availableSlots`/`deliveryEnabled` propsami; importuje z `@/lib/delivery-slots` **tylko typ** `AvailableSlot` (typy znikają w kompilacji, bezpieczne). NIE wywołuje utila odczytu.

Złamanie którejkolwiek warstwy = **cicha awaria hydratacji wyspy klienckiej** (bez błędu w konsoli). Patrz memory `use-server-turbopack-gotcha` i nagłówek `delivery-slots.ts:6`.

### Wzór odczytu frontendowego (NFR1 — izolacja tenant)
Każdy odczyt na froncie idzie przez `overrideAccess: true` **z ręcznym `where { tenant }`** — plugin multi-tenant NIE pilnuje publicznych zapytań (pilnuje panelu/zapisu). Wzór: `getCatalog` w `src/lib/shop.ts:38` i `getCartSnapshot` w `src/lib/cart.ts:35`. **Brak filtra `tenant` = wyciek cross-tenant.** Tu dotyczy obu odczytów: `delivery-slots` i `delivery-date-exceptions`.

### Mapowanie dokumentów na `DeliverySlot`
Pola kolekcji `delivery-slots` (z S2.1, kontrakt SPIKE-S2) mapują się 1:1 na typ `DeliverySlot` (`delivery-slots.ts:29`). **Jedyna różnica:** `reservedCount` NIE jest polem kolekcji — to argument wejściowy funkcji. W tej historii ustaw `reservedCount: 0` (AC#3). Gdy `capacity` przyjdzie jako `number | null/undefined` z bazy — zabezpiecz (`?? 0`), bo funkcja robi `Math.floor`.

### Kodowanie wyboru: potrzebne ID + DATA
Jeden cykliczny slot (`id`) generuje wiele wystąpień (różne `date`). Do późniejszej walidacji (S2.3) i snapshotu (S2.4) `placeOrder` musi znać **oba**: `id` slotu i konkretną `date` (`YYYY-MM-DD`). Dlatego value opcji selecta koduje parę (`` `${id}__${date}` ``), a stan/parametr akcji to `{ id, date }`. Nie wystarczy samo `id`.

### Etykieta dnia tygodnia (PL)
`AvailableSlot.weekday` to `0–6` wg JS `getDay()` (**0 = niedziela**). Stała tablica PL: `['niedz.','pon.','wt.','śr.','czw.','pt.','sob.']` indeksowana `weekday`. Datę formatuj prosto z `date` (`YYYY-MM-DD` → `DD.MM.YYYY` lub `DD.MM`). Trzymaj formatowanie lekkie/inline w komponencie (lub mały helper) — bez nowych zależności dat.

### Stan UI i `disabled` submitu
Dotychczasowy guard: `disabled={belowMin || submitting}`. Dodaj człon dla slotu: gdy `deliveryEnabled` i brak `selectedSlot` → submit zablokowany (AC#7). Gdy `deliveryEnabled` i pusta lista → render komunikatu zamiast selecta + zablokowany submit. Gdy `!deliveryEnabled` → zero zmian w zachowaniu submitu (checkout jak dziś, O8).

### Środowisko / workflow
- **Postgres = Docker** (`od-sasiada-pg`, `postgres:17`, port 5432, baza `od_sasiada`) — NIE Homebrew. Jeśli daemon padł: `open -a Docker`, poczekaj aż `docker info` przejdzie, potem `docker start od-sasiada-pg`.
- **Biome** jedyny lint/format: single quotes, bez średników, `organizeImports`/`useSortedKeys`/`useSortedProperties` ON — pisz w dowolnej kolejności i odpal `pnpm lint:fix`.
- **Testy:** integracyjne `tests/integration/*.integration.test.ts` (live DB, `fileParallelism: false`, jedna instancja Payload, timeout 60s); unit `tests/unit/*.test.ts`. Alias `@/` działa w testach.
- Pieniądze (`remaining`/`capacity`) to liczniki miejsc, nie kwoty — nie dotyczy reguły groszy; ceny w `CartView` nadal przez `formatPLN`.

- ❌ Nie liczyć/filtrować slotów po stronie klienta — `CartView` tylko renderuje gotową listę z propsów.
