# Story 3.3: Render zdjęcia (katalog + szczegół zamówienia) z fallbackiem wariant→produkt→placeholder

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

Jako **klient**,
chcę **widzieć zdjęcie produktu na katalogu `/[tenant]` i miniaturę w szczegółach zamówienia**,
aby **wiedzieć, co kupuję** (decyzja D3; ryzyka R-S3.2 izolacja, R-S3.3 wydajność, R-S3.6 reuse fallbacku).

## Acceptance Criteria

1. **Fallback wyboru zdjęcia: wariant → produkt → placeholder** (D3), zaimplementowany w **jednej czystej funkcji** `resolveProductImage(variantImage, productImage)` (reuse + testowalność, R-S3.6).
2. **Karta produktu (`Catalog.tsx`)** renderuje zdjęcie przez **`next/image`** (responsywne `sizes`, lazy-load, `alt` z mediów). Zdjęcie zmienia się wg wybranego wariantu (fallback do produktu); **placeholder** gdy brak na obu poziomach.
3. **Szczegół zamówienia — front klienta** (`/[tenant]/moje-zamowienia/[id]`): miniatura pozycji = zdjęcie **wybranego wariantu**, fallback do produktu, placeholder gdy brak. (Panel admina — patrz Q1; może być prostszy/odłożony.)
4. **Izolacja (R-S3.2):** odczyt mediów na froncie z `overrideAccess: true` + **ręczny `where { tenant }`** — render nie może przeciekać mediów cross-tenant (defense-in-depth, mimo walidacji zapisu z S3.2).
5. **`next/image` skonfigurowany dla Vercel Blob** — `images.remotePatterns` w `next.config` dopuszcza domenę bloba (`*.public.blob.vercel-storage.com`); URL-e lokalne (dev, `/api/media/...`, same-origin) działają bez zmian.
6. **Bez regresji wydajności (NFR8/R-S3.3)** — użyj wariantu `sharp` (`card`/`thumbnail`) i `sizes`; LCP listy pod kontrolą. **Katalog/koszyk działają jak dziś** (`Catalog.tsx` to client island + `cart-store`). `pnpm lint`/`pnpm test`/`pnpm dev` zielone.

## Dev Notes

### Co to za historia (zakres ścisły)
S3.3 = **render** zdjęć (katalog + szczegół zamówienia) + **fallback** + konfiguracja `next/image`. Buduje na polach `heroImage` z S3.2. **Poza zakresem:** filtrowanie po kategorii (S3.6), zarządzanie mediami/kategoriami (S3.1/S3.4/S3.5).

### Fallback w JEDNYM helperze (R-S3.6)
Logika „wariant → produkt → placeholder" pojawia się w ≥2 miejscach (karta produktu, miniatura zamówienia). Trzymaj ją w jednej czystej funkcji `resolveProductImage` (`src/lib/product-image.ts`, bez I/O/`'use server'`) — testowalnej i bez rozjazdu. Komponenty tylko ją wołają na już-rozwiązanych danych `image`.

### Granica server/client + skąd `image`
- **Wybór zdjęcia jest po stronie klienta** (zależy od wybranego wariantu w `Catalog.tsx`), więc KAŻDY wariant i produkt niosą swoje `image` w propsach (`CatalogProduct`/`CatalogVariant`). Rozwiązanie (`resolveProductImage`) działa w komponencie klienckim — funkcja czysta, import bezpieczny.
- **Dane `image` produkuje serwer** (`getCatalog` w `shop.ts`, plain server util) — czyta `media` tenant-scoped i wstrzykuje do typów. `next/image` działa w client island bez problemu (URL + alt + wymiary).
- `money.ts`/`product-image.ts` są **client-safe** (bez server-only importów) — zgodnie z regułą „czyste typy/util client-safe" (patrz [[use-server-turbopack-gotcha]]).

### Izolacja przy renderze (R-S3.2) — defense in depth
Mimo że S3.2 waliduje tenant-match przy zapisie `heroImage`, ścieżka odczytu i tak czyta media z `overrideAccess: true` → **musi mieć `where { tenant }`**. Inaczej spreparowany/odziedziczony rekind mógłby ujawnić cudze medium. To ten sam wzór co `getCatalog`/`getCartSnapshot` (ręczny filtr tenant na publicznych odczytach).

### Stan plików (UPDATE)
- **`src/lib/shop.ts` `getCatalog`** (`#38`): dziś czyta produkty + warianty (`depth: 0`), mapuje do `CatalogProduct`/`CatalogVariant` bez zdjęć. Dodaj odczyt mediów (tenant-scoped) + `image`. Zachowaj filtr `_status: published` + `tenant`.
- **`src/lib/money.ts`**: czyste typy — rozszerz `CatalogVariant`/`CatalogProduct` o `image`; dodaj `ProductImage`.
- **`src/components/shop/Catalog.tsx`** (client): `ProductCard` — dodaj render `next/image` + placeholder; reszta (cart-store, warianty, ceny) BEZ ZMIAN.
- **`moje-zamowienia/[id]/page.tsx`** (server): dodaj miniaturę pozycji; NIE ruszać `where { id, customer, tenant }` (IDOR, test `order-detail-idor`).
- **`next.config.ts`**: dziś pusty `nextConfig` opakowany `withPayload`. Dodaj `images.remotePatterns`.

### Wydajność (NFR8 / R-S3.3)
- Użyj wariantu `sharp` adekwatnego do miejsca: karta = `card` (768w) lub `thumbnail` (400²) dla miniatur zamówienia. `next/image` z `sizes` (np. grid → `"(max-width:768px) 50vw, 25vw"`).
- Lazy-load domyślnie; rozważ `priority` na pierwszych kartach above-the-fold. Zmierz LCP listy produktów (cel NFR8).

### Środowisko / workflow
- **`next/image` + Vercel Blob:** wymaga `images.remotePatterns` (absolutne URL-e bloba); lokalnie media to URL same-origin (`/api/media/...`) → działa bez patternu. Bez `BLOB_READ_WRITE_TOKEN` (S3.1) testujesz na dysku lokalnym — render i tak działa.
- **Postgres = Docker** (`od-sasiada-pg`) — NIE Homebrew.
- **Biome** (single quotes, bez średników, `organizeImports` ON) — `pnpm lint:fix`. Pamiętaj `useSortedAttributes` dla JSX (atrybuty `next/image` sortowane automatem).
- **Testy:** unit `tests/unit/*` (czyste — `resolveProductImage`); integracyjne `tests/integration/*` (live DB).

### Czego NIE robić (anti-patterns)
- ❌ Nie duplikuj logiki fallbacku — jeden `resolveProductImage` (R-S3.6).
- ❌ Nie czytaj `media` bez `where { tenant }` na froncie (wyciek cross-tenant, R-S3.2).
- ❌ Nie używaj surowego `<img>` — `next/image` (NFR8: rozmiary/lazy/format).
- ❌ Nie zapomnij `images.remotePatterns` — inaczej obrazy z bloba rzucą błąd `next/image` w produkcji.
- ❌ Nie wprowadzaj server-only importów do `money.ts`/`product-image.ts` (psuje client island).
- ❌ Nie ruszaj warunku izolacji w szczegółach zamówienia (`where { customer, tenant }`) ani logiki koszyka w `Catalog.tsx`.
- ❌ Nie „snapshotuj" zdjęcia jako historycznego (to live miniatura — patrz Q2); nie myl z B1 (cena/nazwa są snapshotowane, zdjęcie nie).

## Open Questions (do potwierdzenia przed/po dev)
- **Q1 — Miniatura w PANELU admina.** Front klienta jest prosty (server component). Miniatura w widoku zamówienia w panelu wymaga custom admin cell/field component (cięższe). Rekomendacja: front teraz; panel — minimalnie/odłożyć, jeśli wykracza poza budżet (potwierdź zakres).
- **Q2 — Zdjęcie live vs snapshot.** Rekomendacja: miniatura zamówienia rozwiązywana **live** z bieżącego `heroImage` (zdjęcie nie jest snapshotowane jak cena/nazwa). Zmiana zdjęcia produktu zmieni miniaturę starego zamówienia — akceptowalne dla miniatury. Potwierdź (alternatywa: snapshot URL-a do zamówienia — większy zakres, raczej nie).
- **Q3 — Host Vercel Blob w `remotePatterns`.** Dokładny hostname (`*.public.blob.vercel-storage.com`) potwierdź po sprovisionowaniu tokenu (S3.1). Do tego czasu render działa na URL-ach lokalnych (same-origin).
- **Q4 — Placeholder.** Rekomendacja: box CSS (brak `public/`), bez assetu. Alternatywa: dodać `public/placeholder.svg` + `next/image`. Potwierdź.
