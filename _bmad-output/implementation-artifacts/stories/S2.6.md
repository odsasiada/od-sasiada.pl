# Story 2.6: PL-izacja i wzbogacenie maili (slot dostawy + kontakt dostawcy)

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

Jako **operator platformy**,
chcę **treści maili (potwierdzenie + zmiana statusu) po polsku oraz z terminem dostawy i telefonem kontaktowym dostawcy**,
aby **komunikacja z klientem była po polsku i kompletna** (domknięcie pętli „wiem co się dzieje").

## Acceptance Criteria

1. **Treść maila potwierdzającego (`sendOrderConfirmation`) po polsku** — nagłówek, opis, etykiety (pozycje, suma, adres), temat. Kwoty nadal w formacie `X,XX zł`.
2. **Treść maila statusowego (`sendStatusChange`) po polsku** — nagłówek, zdanie o zmianie statusu, temat; etykieta statusu z PL `ORDER_STATUS_LABELS`.
3. **Etykiety statusów PL** — `ORDER_STATUS_LABELS` w `src/ecommerce/order-status.ts` po polsku (dziś po ang.). Skutek spójny WSZĘDZIE, gdzie używane: opcje pola `status` w panelu, strony frontu (`moje-zamowienia`), komunikat błędu przejścia, maile.
4. **Mail potwierdzający zawiera termin dostawy** ze snapshotu `deliverySlot` zamówienia (S2.4) — gdy zamówienie ma slot; gdy nie ma (O8) → sekcja pominięta, bez błędu.
5. **Mail potwierdzający zawiera telefon kontaktowy dostawcy** (`tenant.settings.contactPhone`) — gdy ustawiony; gdy brak → sekcja pominięta.
6. **Best-effort zachowane (NFR4)** — błąd SMTP/odczytu logowany, NIE blokuje operacji zapisu zamówienia; brak `customerEmail` → brak wysyłki (zachowany guard).
7. **Testy zaktualizowane i zielone** — `tests/unit/order-emails.test.ts` (asercje treści ang. → PL + slot/telefon), ewentualnie `order-status.test.ts`; `pnpm lint` + `pnpm test` bez regresji; `pnpm dev` startuje.

## Dev Notes

### Co to za historia (zakres ścisły)
S2.6 = **treść i język komunikacji** — domknięcie EPIC-2. Zmieniamy **CO** zawierają maile (PL + slot + kontakt) i etykiety statusów. **NIE** zmieniamy **kiedy** maile lecą (to S2.5), ani mechanizmu snapshotu slotu (to S2.4), ani automatu stanów.

**Poza zakresem (świadomie):**
- **S2.5** — kiedy mail statusowy leci (milestone'y). S2.6 zakłada, że `sendStatusChange` jest już zawężony.
- **S2.4** — snapshot `deliverySlot` na zamówieniu. S2.6 tylko go CZYTA do treści.
- PL-izacja UI frontu (`CartView`, nagłówki `moje-zamowienia` typu „Your cart"/„Order details") — **NIE** w tej historii (osobny zakres; tu PL-izujemy maile + etykiety statusów, które przy okazji PL-izują status na froncie). Patrz Q3.

### Stan obecny plików (UPDATE) — co zachować
- **`src/ecommerce/order-emails.ts`**: dwa buildery `(payload, doc)`. `sendOrderConfirmation` ma `itemsTable`, `addressBlock`, `zl()` (groszowe „X,XX zł"); `sendStatusChange` używa `ORDER_STATUS_LABELS`. Oba mają guard `if (!doc.customerEmail) return`. Treść dziś po ANGIELSKU. **Zachowaj:** guard `customerEmail`, `zl()` formatowanie, strukturę builderów; zmień teksty na PL + dołóż slot/telefon.
- **`src/ecommerce/order-status.ts`**: `ORDER_STATUS_LABELS` (ang.) — jedyne źródło etykiet. Zmieniamy tylko wartości `label`. `value`/`ORDER_STATUS_SEQUENCE`/`isAllowedTransition`/`shouldEmailStatusChange` (S2.5) BEZ ZMIAN.
- **Hook `notifyOnOrderChange`** (`orders.ts`) — BEZ ZMIAN (wyzwalanie to S2.5). S2.6 dotyka tylko builderów treści.

### ⚠️ Ripple etykiet (AC#3) — sprawdź WSZYSTKICH konsumentów
`ORDER_STATUS_LABELS` jest importowane w wielu miejscach. Zmiana na PL zmienia tekst:
- panel: opcje pola `status` (`orderStatusField` w `order-status.ts`) → PL,
- front: `moje-zamowienia/page.tsx` i `moje-zamowienia/[id]/page.tsx` (`ORDER_STATUS_LABELS[...]`) → PL badge,
- `validateStatusTransition` (`orders.ts`) — komunikat błędu → PL,
- maile (Task 2/3).
- **TESTY:** `tests/unit/order-emails.test.ts` asertuje `'Confirmed'` (subject/html) — MUSI zostać zaktualizowane na PL, inaczej regresja. Sprawdź też `order-status.test.ts`.

### Dane do treści — skąd je wziąć
- **Slot:** ze snapshotu `doc.deliverySlot` (S2.4: `date`, `windowStart`, `windowEnd`, `label`). Preferuj gotowe `label` (jedno źródło formatowania, `formatSlotLabel` z S2.4); fallback złóż z `date` + `windowStart–windowEnd`. Render warunkowo (O8: brak slotu → brak sekcji).
- **Telefon dostawcy:** `tenant.settings.contactPhone`. Builder ma `payload` → `payload.findByID({ collection: 'tenants', id: tenantIdOf(doc), depth: 0, disableErrors: true })`. `doc.tenant` bywa `number | { id }` — znormalizuj (wzór `idOf` w `actions.ts`/`cart.ts`). Best-effort: błąd/brak → pomiń sekcję, NIE wywracaj maila. (Wzór mapowania `contactPhone` już jest w `getTenantBySlug`, `src/lib/shop.ts:25`.)

### NFR4 — best effort
Buildery wołane z `afterChange` przez `.catch(log)` (`orders.ts`). Dodatkowy `findByID` tenanta NIE może rzucać w górę — użyj `disableErrors: true` i obsłuż `null`. Brak `customerEmail` nadal = brak wysyłki.

### Język / styl treści
- Teksty maili po polsku (i18n `pl`); identyfikatory/kod po angielsku. Kwoty przez istniejące `zl()` („X,XX zł", przecinek dziesiętny).
- Trzymaj prostą strukturę HTML inline-style (jak dziś) — bez nowych zależności/szablonów.

### Środowisko / workflow
- **Biome** jedyny lint/format (single quotes, bez średników, `organizeImports` ON) — `pnpm lint:fix`.
- **Testy:** unit `tests/unit/*.test.ts` (czyste, stub `payload`); integracyjne `tests/integration/*` (live Docker `od-sasiada-pg`, `fileParallelism:false`).
- Zmiana etykiet nie zmienia schematu (`label` to nie `value`) → `pnpm payload:types` zwykle niepotrzebne; jeśli `OrderDoc` mapujesz z `Order` (slot/tenant), upewnij się, że typy z S2.4 są zregenerowane.

### Czego NIE robić (anti-patterns)
- ❌ Nie zmieniaj `value`/kluczy `ORDER_STATUS_LABELS` (tylko `label`) — `value` to kontrakt automatu/DB.
- ❌ Nie ruszaj wyzwalania maili (S2.5: `shouldEmailStatusChange`, `notifyOnOrderChange`).
- ❌ Nie zmieniaj snapshotu slotu / kolekcji (S2.4) — tylko czytasz `doc.deliverySlot`.
- ❌ Nie czyń wysyłki/`findByID` blokującymi — best-effort, `disableErrors`, guard `customerEmail` (NFR4).
- ❌ Nie zmieniaj formatowania kwot (`zl()`/grosze).
- ❌ Nie zostawiaj testów asertujących ang. etykiety (regresja) — zaktualizuj.
- ❌ Nie PL-izuj przy okazji UI frontu poza etykietami statusów (poza zakresem; Q3).
