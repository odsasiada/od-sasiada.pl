## Story

Jako **dostawca**,
chcę **w panelu widzieć dzienne obłożenie slotów (zajęte / limit per slot i dzień)**,
aby **planować rozwóz i widzieć, które terminy są bliskie wypełnienia** (decyzja O4; widok pomocniczy, NIE egzekwujący).

## Acceptance Criteria

1. **Widok panelu (read-only) pokazuje obłożenie per wystąpienie slotu** (slot + konkretny dzień) na horyzoncie dostępnych terminów: „zajęte / `capacity`" (np. `3 / 10`).
2. **Obłożenie = recount aktywnych zamówień z `orders`** (`status not_in ['cancelled']`), grupowane per `(slot, data)` — **to samo źródło prawdy co S2.7** (bez licznika inkrementalnego, bez driftu). Reużyj logiki/recountu z S2.7, nie pisz drugiej.
3. **Tenant-scoped (NFR1).** Dostawca widzi WYŁĄCZNIE własne sloty i obłożenie; `platform-admin` widzi wszystko (lub z selektorem tenanta). Brak danych cross-tenant.
4. **Wyłącznie odczyt.** Widok niczego nie zapisuje ani nie egzekwuje — egzekwowanie capacity to S2.7. Brak zmian w `placeOrder`/hookach.
5. **Spójność z listą klienta.** Wystąpienia i ich liczenie zgodne z `computeAvailableSlots` + recountem z S2.2/S2.7 (te same daty/okna/`remaining`).
6. **Brak slotów u tenanta (O8) → pusty/informacyjny stan**, bez błędu. `pnpm dev`/`pnpm build` startują; `pnpm lint` + `pnpm test` bez regresji.

## Dev Notes

### Co to za historia (zakres ścisły)
S2.9 = **read-only widok obłożenia** dla dostawcy. NIE wprowadza żadnego egzekwowania — to czysta prezentacja danych, które już produkuje S2.7. Wartość: planowanie rozwozu / podgląd zapełnienia. **PARK**, bo egzekwowanie (clou O4) żyje już w S2.7, a sam widok jest „nice to have".

**Poza zakresem:** jakakolwiek mutacja, egzekwowanie capacity (S2.7), zmiana modelu slotów (S2.1) czy snapshotu (S2.4).

### Fundament — reużyć z S2.7 (krytyczne, antidrift)
Obłożenie MUSI pochodzić z tego samego recountu co enforcement (S2.7): aktywne zamówienia per `(slot, data)`, `status not_in ['cancelled']`. **Nie implementuj drugiego, równoległego liczenia** — wydziel/reużyj helper z S2.7 (`getAvailableDelivery`/recount w `delivery-slots-read.ts`). Inaczej widok i egzekwowanie się rozjadą. `cancelled` automatycznie nie liczy się do `reserved` (filtr), spójnie z regułą cyklu życia.

### Zależność od kształtu `deliverySlot` (S2.7/S2.4)
Recount per-occurrence wymaga, by `orders` miały `deliverySlot.slot` + `deliverySlot.date` (wprowadza S2.7; granularność per-occurrence — patrz S2.7 Q1). S2.9 czyta dokładnie te pola. **Buduj dopiero po S2.7** (i S2.1). Jeśli S2.7 zmieni kształt/nazwy pól — dostosuj zapytanie.

### Custom admin component (Payload v3) — uwaga
- Payload v3 rejestruje custom komponenty/widoki przez `admin.components` w configu; wymaga wpisu w generowanym `src/importMap.js` (`pnpm payload:importMap`). **Nie edytuj `importMap.js` ręcznie.**
- Preferuj **server component** (czyta przez Local API serwerowo) — brak granicy hydratacji jak na froncie sklepu, ale i tak trzymaj odczyt po stronie serwera.
- Kontekst tenanta w panelu: zalogowany user (`users`) + `multiTenantPlugin` (rola `platform-admin` widzi wszystko via `userHasAccessToAllTenants`). Dokładny sposób pobrania „bieżącego tenanta admina" w custom view — do potwierdzenia względem API pluginu (Q2).
- **Dokładny punkt wpięcia (osobny widok vs komponent na dashboard vs kolumna)** i API custom view różnią się między wersjami — zweryfikuj w zainstalowanym `payload@^3.85.1`, nie zgaduj (Q3).

### Izolacja (NFR1)
Każdy odczyt obłożenia z `overrideAccess: true` MUSI mieć `where { tenant }` (+ filtr `deliverySlot.slot`/`date`/status). Dla `platform-admin` — albo brak filtra tenanta (wszyscy), albo selektor. Brak filtra dla zwykłego dostawcy = wyciek cross-tenant. Test izolacji obowiązkowy.

### Środowisko / workflow
- **Postgres = Docker** (`od-sasiada-pg`) — NIE Homebrew.
- **Custom admin component → `pnpm payload:importMap`** (i `payload:types` przy zmianie pól). Build robi automatycznie; lokalnie ręcznie.
- **Biome** (single quotes, bez średników, `organizeImports` ON) — `pnpm lint:fix`.
- **Testy:** integracyjne `tests/integration/*` (live DB, `fileParallelism:false`).

### Czego NIE robić (anti-patterns)
- ❌ Nie pisz drugiego liczenia obłożenia — reużyj recount z S2.7 (jedno źródło prawdy).
- ❌ Nie egzekwuj/zmieniaj capacity tutaj (to S2.7) — widok jest read-only.
- ❌ Nie czytaj obłożenia bez `where { tenant }` (wyciek cross-tenant, NFR1).
- ❌ Nie edytuj `importMap.js`/`payload-types.ts` ręcznie (generowane).
- ❌ Nie buduj przed S2.7/S2.1 (brak pól `deliverySlot.*` do recountu).
- ❌ Nie flipuj statusu na ready-for-dev bez świadomej decyzji (PARK).
