# Story 2.5: Maile statusowe tylko na milestone'ach (zawężenie istniejącego hooka)

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

Jako **klient**,
chcę **dostawać maila tylko przy istotnej zmianie statusu zamówienia (milestone'y „do przodu", bez spamu przy cofnięciu)**,
aby **wiedzieć co się dzieje z zamówieniem bez dzwonienia do dostawcy** (decyzje O5/O6; ryzyko R-S2.3).

> ⚠️ To **ZAWĘŻENIE** istniejącego `sendStatusChange`, który DZIŚ wysyła maila na KAŻDEJ zmianie statusu (podpięty w `afterChange` od S1.5). Realna praca = ograniczenie wyzwalania do milestone'ów + cisza przy cofnięciu + brak duplikatów. **NIE** „dodanie maili od zera".

## Acceptance Criteria

1. **Mail statusowy wychodzi WYŁĄCZNIE na milestone'ach (O5):** docelowy status ∈ `{ confirmed, out_for_delivery, delivered, cancelled }`. Przejścia do `new` (w tym reaktywacja `cancelled→new`) i do `preparing` **NIE** wysyłają maila.
2. **Cisza przy cofnięciu (O6/R-S2.3).** Cofnięcie w sekwencji (np. `delivered→out_for_delivery`, `out_for_delivery→confirmed`) → **bez maila**, nawet gdy docelowa wartość jest milestone'em. Mail tylko przy ruchu „do przodu" w sekwencji; `cancelled` traktowane jako zdarzenie wysyłkowe (anulowanie z dowolnego niedostarczonego stanu → mail).
3. **Brak duplikatów / brak maila bez realnej zmiany.** Zapis bez zmiany statusu (ta sama wartość docelowa) → brak maila (zachowany warunek `doc.status !== previousDoc.status`).
4. **Błąd SMTP logowany, NIE blokuje update (NFR4).** Zachować obecny wzorzec „best effort" (`.catch(log)`), wysyłka nie wstrzymuje operacji zapisu zamówienia.
5. **Bez zmian treści/wzbogacenia maila.** PL-izacja treści + dodanie slotu/kontaktu dostawcy = **S2.6** (poza zakresem). S2.5 zmienia tylko **kiedy** mail leci, nie **co** zawiera. `sendOrderConfirmation` (create) bez zmian.
6. **Logika decyzji w czystym, testowalnym predykacie** (nie rozsiana po hooku). `pnpm lint` + `pnpm test` zielone bez regresji (`order-status-machine`, `order-emails`), `pnpm dev` startuje.

### Idempotencja / „brak duplikatów" (AC#3)
Warunek `doc.status !== previousDoc.status` jest już w hooku i jest **podzbiorem** predykatu (`from===to → false`), więc zapis bez zmiany statusu nie wyśle maila. **Świadome ograniczenie:** ponowne osiągnięcie tego samego milestone'a po cofnięciu (np. `confirmed→new→confirmed`) wyśle maila ponownie — to zamierzone (realna ponowna zmiana), NIE traktujemy jako duplikat. Nie wprowadzamy pola „ostatnio zmailowany status" (over-engineering; patrz Q1).

### NFR4 — wysyłka nie blokuje
Zachowaj `sendStatusChange(...).catch((err) => req.payload.logger.error(...))`. Mail jest efektem ubocznym `afterChange`; błąd SMTP nie może wywrócić zapisu zamówienia (np. zmiany statusu przez dostawcę w panelu).

### Środowisko / workflow
- **Biome** jedyny lint/format (single quotes, bez średników, `organizeImports` ON) — `pnpm lint:fix`.
- **Testy:** unit `tests/unit/*.test.ts` (czyste, bez DB) — tu predykat; integracyjne `tests/integration/*.integration.test.ts` (live Docker `od-sasiada-pg`, `fileParallelism:false`).
- Nie zmieniasz kolekcji/pól → `pnpm payload:types` niepotrzebne.

### Czego NIE robić (anti-patterns)
- ❌ Nie PL-izuj ani nie wzbogacaj treści `sendStatusChange` (S2.6).
- ❌ Nie ruszaj gałęzi `create`/`sendOrderConfirmation` ani `isAllowedTransition`.
- ❌ Nie czyń wysyłki blokującą — zostaw `.catch` (NFR4).
- ❌ Nie hardkoduj kolejności statusów — reużyj `ORDER_STATUS_SEQUENCE`.
- ❌ Nie dodawaj pola „last emailed status" do kolekcji (niepotrzebne; patrz Q1).
- ❌ Nie wysyłaj na `preparing` ani na reaktywację `cancelled→new` (nie są milestone'ami O5).
