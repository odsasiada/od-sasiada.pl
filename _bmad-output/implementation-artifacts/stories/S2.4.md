# Story 2.4: Snapshot slotu w zamówieniu (panel + front read-only + mail)

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

Jako **dostawca i klient**,
chcę **widzieć wybrany termin dostawy w szczegółach zamówienia (panel + front read-only) oraz w mailu potwierdzającym**,
aby **termin był wiążącym snapshotem zamówienia, odpornym na późniejsze zmiany konfiguracji slotów** (decyzja B1; SPIKE-S2 „zamówienia to referencje").

## Acceptance Criteria

1. **Slot zrzucony jako SNAPSHOT do zamówienia przy tworzeniu** (analogicznie do `shippingAddress` i `productNameSnapshot`/`unitPriceSnapshot`). Snapshot zawiera co najmniej: **datę wystąpienia** (`YYYY-MM-DD`), **okno** (`windowStart`, `windowEnd` jako `HH:mm`) i **etykietę** czytelną dla człowieka (np. „pon. 23.06.2026, 08:00–12:00"). Wypełniany w `placeOrder` z **zwalidowanego** wystąpienia (S2.3), nie z surowego ciała żądania.
2. **Odporność na zmianę configu (B1).** Po złożeniu zamówienia edycja/usunięcie rekordu w `delivery-slots` **nie zmienia** historycznego terminu w zamówieniu (snapshot = kopia wartości, nie live-join). Potwierdzone testem.
3. **Widoczny w panelu dostawcy** (pola snapshotu `readOnly` w override kolekcji `orders`, automatycznie w adminie; izolacja tenant gratis przez plugin).
4. **Widoczny na froncie klienta — read-only, izolacja `where { customer, tenant }` + `overrideAccess: true`** (wzór S1.6) — w szczegółach zamówienia `/[tenant]/moje-zamowienia/[id]`. Sekcja renderowana tylko gdy snapshot istnieje (stare zamówienia bez slotu / tenant bez okien → brak sekcji, bez błędu).
5. **Dołączony do maila potwierdzającego** (`sendOrderConfirmation` w `order-emails.ts`) jako dodatkowy blok obok adresu. (Pełna PL-izacja treści maila = S2.6; tu dokładamy dane slotu.)
6. **Spójne formatowanie etykiety** — jedna czysta funkcja formatująca etykietę slotu (dzień PL + data + okno) współdzielona przez snapshot/front/mail (i, jeśli S2.2 zinline'owało, skonsolidowana z UI). Zero rozjazdu między listą wyboru a snapshotem.
7. **`pnpm payload:types` przechodzi** (nowe pola na typie `Order`), **`pnpm lint` + `pnpm test` zielone bez regresji** (w tym `orders-isolation`, `order-detail-idor`), `pnpm dev` startuje, checkout end-to-end działa (ze slotem zapisuje snapshot; bez okien — bez zmian).

## Dev Notes

### Co to za historia (zakres ścisły)
S2.4 = **snapshot wybranego slotu na zamówieniu + jego prezentacja** (panel, front read-only, mail). To domknięcie pętli „wiem, kiedy dojedzie": termin staje się wiążącym, niezmiennym elementem zamówienia.

**Poza zakresem (świadomie):**
- **S2.5** — zawężenie maili statusowych (milestone'y, bez spamu przy cofnięciu). Tu NIE ruszamy `sendStatusChange`/`notifyOnOrderChange`.
- **S2.6** — pełna PL-izacja treści maili (potwierdzenie + status) + dane kontaktowe dostawcy. Tu dokładamy tylko **dane slotu** do maila potwierdzającego.
- **S2.7** — capacity/wyścig (recount, transakcja). Tu zakładamy, że S2.7 już istnieje; koordynujemy kształt pola `deliverySlot` (Q1).

### KRYTYCZNE: kolejność i koordynacja z S2.7 (R-sequencing)
Build order to S2.3 → **S2.7** → **S2.4**. S2.7 (recount capacity) liczy aktywne zamówienia na **wystąpienie** slotu, więc zamówienie musi nieść referencję slotu **i datę wystąpienia** zapisane przy `create`. To znaczy, że część persistencji (`deliverySlot.slot` + `deliverySlot.date`) mogła już zostać wprowadzona w S2.7.
- **Przed implementacją sprawdź aktualny `src/ecommerce/orders.ts` i `placeOrder`**: czy S2.7 dodał już pole/relację slotu i czy zapisuje datę. Jeśli tak — S2.4 **rozszerza** ten sam blok o pola prezentacyjne (`windowStart/windowEnd/label`) i NIE tworzy drugiego, konkurencyjnego pola.
- Capacity jest **per-occurrence** (limit na JEDNO wystąpienie = slot + konkretna data; SPIKE-S2 (a)). Dlatego recount S2.7 musi filtrować po `deliverySlot.slot` **oraz** `deliverySlot.date` — `date` w snapshotcie jest współdzielone. Trzymaj `date` jako pojedyncze, queryable pole (dot-path w Payload: `where { 'deliverySlot.date': ... }`).

### Wzorce snapshotu — DOKŁADNIE do naśladowania
- **Pola snapshotu pozycji** (`SNAPSHOT_FIELDS`, `orders.ts:26`): `readOnly`, kopiowane przy tworzeniu, „immune to later price changes". Slot = analogiczny mechanizm, ale wartości pochodzą ze **zwalidowanego wystąpienia** (nie z lookupu produktu).
- **`shippingAddress`** (zapisywany w `placeOrder`, `actions.ts:135`): grupa pól adresu zrzucana z `contact`. `deliverySlot` to bliźniaczy blok — zrzucany z `matched: AvailableSlot`.
- **Gdzie populować:** w `placeOrder` (gdzie żyje zwalidowane wystąpienie), NIE w hooku `beforeValidate` (tam nie ma kontekstu wyboru klienta — line-item snapshot da się odtworzyć z `items`, slot NIE). To różnica względem `populateSnapshotAndNumber`.

### Integracja z walidacją S2.3 (mały, ważny refaktor)
S2.3 wprowadził `validateChosenSlot(tenantId, chosen) → { ok } | { ok:false, error }`. S2.4 potrzebuje **dopasowanego wystąpienia** do snapshotu → zmień zwracany typ na `{ ok: true; slot: AvailableSlot } | { ok: false; error: string }` (zwróć element znaleziony przez membership-check). `placeOrder` używa `result.slot` do zbudowania snapshotu. To jedyne źródło `windowStart/windowEnd/date/weekday` — nie czytaj ich ponownie z DB (uniknięcie wyścigu/driftu; zwalidowany zbiór już je ma).

### Etykieta — jedna czysta funkcja (AC#6)
`formatSlotLabel` w `delivery-slots.ts` (czysty moduł): używana przez (a) snapshot w `placeOrder`, (b) front detail, (c) mail, oraz docelowo (d) `CartView` z S2.2. Trzyma nazwy dni PL i format daty w jednym miejscu. `weekday` 0–6 (0=niedziela). Bez bibliotek dat (zgodnie ze SPIKE-S2).

### Front detail — co zachować (NFR1 / IDOR)
`moje-zamowienia/[id]/page.tsx` już izoluje: `where { and: [{ id }, { customer }, { tenant }] }` + `overrideAccess: true` + `notFound()` przy braku. **Nie ruszać tego warunku** — to ochrona IDOR potwierdzana testem `order-detail-idor`. Dodajesz tylko render bloku slotu (read-only). Strona jest server-component (brak granicy hydratacji do złamania).

### Panel
Pola `deliverySlot` w override są `readOnly` → admin pokazuje je automatycznie; izolacja tenant przez plugin (dostawca widzi tylko swoje zamówienia). Rozważ `defaultColumns`/grupowanie tylko jeśli trywialne — nie wymagane przez AC.

### Mail
`sendOrderConfirmation` (`order-emails.ts:49`) jest dziś po angielsku — **NIE PL-izuj całości tutaj** (S2.6). Dodaj blok slotu (jak `addressBlock`) korzystając ze snapshotu. `OrderDoc` to lokalny typ w tym pliku — rozszerz o `deliverySlot?`. Wysyłka jest „best effort" (błędy logowane, nie blokują) — zachowaj.

### Środowisko / workflow
- **Postgres = Docker** (`od-sasiada-pg`, `postgres:17`, port 5432, baza `od_sasiada`) — NIE Homebrew. Daemon padł: `open -a Docker` → poczekaj na `docker info` → `docker start od-sasiada-pg`.
- **Po zmianie pól kolekcji ZAWSZE `pnpm payload:types`** (build robi automatycznie; lokalnie ręcznie). Nie edytować `payload-types.ts`/`importMap.js`.
- **Biome** jedyny lint/format: single quotes, bez średników, `organizeImports`/`useSortedKeys`/`useSortedProperties` ON — pisz dowolnie, odpal `pnpm lint:fix`.
- **Testy:** integracyjne `tests/integration/*.integration.test.ts` (live DB, `fileParallelism: false`, jedna instancja Payload, timeout 60s). Alias `@/` działa.
- **Pieniądze** w groszach + `formatPLN` — slot nie dotyczy kwot; w mailu kwoty już przez `zl()`.

### Czego NIE robić (anti-patterns)
- ❌ Nie rób live-join slotu w prezentacji — snapshot to KOPIA przy `create` (B1; inaczej zmiana configu przepisze historię).
- ❌ Nie czytaj okna/daty ponownie z `delivery-slots` w `placeOrder` — użyj zwalidowanego `AvailableSlot` z S2.3 (jedno źródło, brak wyścigu/driftu).
- ❌ Nie duplikuj pola `deliverySlot`, jeśli S2.7 już je dodało dla recountu — uzgodnij jeden blok (Q1).
- ❌ Nie populuj slotu w `beforeValidate` (brak kontekstu wyboru) — rób to w `placeOrder`.
- ❌ Nie ruszaj `sendStatusChange`/automatu stanów (S2.5) ani nie PL-izuj całego maila (S2.6).
- ❌ Nie zmieniaj warunku `where { customer, tenant }` na froncie (ochrona IDOR — test `order-detail-idor`).
- ❌ Nie duplikuj formatowania etykiety — `formatSlotLabel` jest jedynym źródłem (AC#6).
- ❌ Nie edytuj `payload-types.ts` ręcznie — regeneruj.
