# Story 3.5: CRUD kategorii w panelu — izolacja tenant (weryfikacja + test)

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

Jako **dostawca**,
chcę **w panelu zarządzać (tworzyć/edytować/usuwać/listować) wyłącznie własnymi kategoriami**,
aby **mieć taksonomię izolowaną od innych dostawców** (NFR1; ryzyko R-S3.4).

## Acceptance Criteria

1. **Dostawca CRUD-uje tylko własne kategorie** — tworzy/edytuje/usuwa kategorie swojego tenanta; nie może modyfikować cudzych.
2. **Dostawca B nie widzi kategorii A** — lista w panelu pokazuje tylko kategorie B; bezpośredni odczyt rekordu A przez Local API z `overrideAccess: false` w kontekście B → pusty/odrzucony (404/brak).
3. **`platform-admin` widzi wszystkie** kategorie wszystkich tenantów (`userHasAccessToAllTenants`).
4. **Izolacja potwierdzona testem integracyjnym** (wzór `orders-isolation`), dołączonym do regresji bezpieczeństwa. Bez regresji: `pnpm test`/`pnpm lint`/`pnpm dev` zielone.

## Dev Notes

### Co to za historia (zakres ścisły)
S3.5 to przede wszystkim **weryfikacja izolacji + test regresyjny**, nie budowa od zera. Izolacja kategorii w panelu jest **out-of-the-box z `multiTenantPlugin`** (jak `products`/`media`/`orders`). Realna praca: napisać dowód (test integracyjny) i — tylko jeśli test wykaże lukę — dostroić `access`. **Poza zakresem:** filtrowanie katalogu po kategorii (S3.6), tworzenie kolekcji (S3.4).

### Dlaczego to działa „za darmo" (kontekst SPIKE-A)
SPIKE-A potwierdził na zamówieniach, że `multiTenantPlugin` pilnuje `update`/widoczności per-tenant — własny `access.update` po tenant okazał się zbędny. Te same gwarancje dotyczą każdej kolekcji wpiętej w `multiTenantPlugin.collections`, więc `categories` dziedziczy izolację panelu. Ta historia po prostu to **udowadnia** dla kategorii (tani guard regresji NFR1).

### Wzór testu izolacji (do naśladowania)
`tests/integration/orders-isolation.integration.test.ts` (z SPIKE-A) jest dokładnym precedensem: tworzy dane dwóch tenantów, symuluje kontekst admina tenanta B (`overrideAccess: false` + `user: adminBUser` z fixtures) i potwierdza brak dostępu cross-tenant (lista pusta / mutacja odrzucona). Fixtures (`tests/setup/fixtures.ts`) dostarczają `tenantA`/`tenantB`/`adminBUser` (`tenant-admin` przypięty do B) oraz `cleanup`.

### Środowisko / workflow
- **Postgres = Docker** (`od-sasiada-pg`) — NIE Homebrew. Daemon padł: `open -a Docker` → `docker info` → `docker start od-sasiada-pg`.
- **Testy:** integracyjne `tests/integration/*.integration.test.ts` (live DB, `fileParallelism:false`, jedna instancja Payload, timeout 60s); alias `@/` działa.
- **Biome** (single quotes, bez średników, `organizeImports` ON) — `pnpm lint:fix`.
- Brak zmian kolekcji/pól → `pnpm payload:types` zwykle niepotrzebne.

### Czego NIE robić (anti-patterns)
- ❌ Nie dodawaj własnego `access` po `tenant` „na wszelki wypadek" — plugin pilnuje; ręczny duplikat może wejść w konflikt (SPIKE-A). Dostrajaj tylko gdy test wykaże lukę.
- ❌ Nie używaj `overrideAccess: true` w teście izolacji (to obchodzi access — testuj z `overrideAccess: false` + `user`).
- ❌ Nie zmieniaj kolekcji `Categories` (to S3.4) ani nie dotykaj filtrowania katalogu (S3.6).
- ❌ Nie filtruj `carts` po `status` (gotcha — nie dotyczy, ale pamiętaj o wzorcach zapytań).
