# S1.0 — Autentykacja klienta (rejestracja / login / logout) per-tenant

Status: ✅ GOTOWE · baseline · EPIC-1

## Historyjka

Jako **klient** chcę **zarejestrować konto, zalogować się i wylogować u konkretnego dostawcy**, aby **móc składać zamówienia jako rozpoznawalny użytkownik, przy czym moje konto u dostawcy A nie daje mi dostępu u dostawcy B** (FR4).

## Kryteria akceptacji

- Klient może się zarejestrować (kolekcja `customers` z autentykacją): konto jest tworzone i klient zostaje od razu zalogowany.
- Klient może się zalogować poprawnymi danymi i wylogować.
- Sesja jest niesiona przez ciasteczko `payload-token` (`httpOnly`, `path: /`, `sameSite: lax`, `secure` w produkcji); logout usuwa ciasteczko.
- E-mail jest **globalnie unikalny** — rejestracja istniejącego e-maila jest odrzucana (komunikat neutralny, bez ujawniania szczegółów).
- **Reguła per-tenant przy logowaniu:** klient należy do dokładnie jednego tenanta; logowanie pod innym tenantem niż jego konto jest **odrzucane** (niedopasowanie tenanta → błąd, brak ustawienia ciasteczka).
- **Izolacja sesji per-tenant przy odczycie:** klient zalogowany u dostawcy A jest traktowany jako **anonimowy** u dostawcy B (sesja zwraca `null`, gdy `tenant` konta ≠ tenant strony).
- Błędne dane logowania zwracają neutralny błąd (bez rozróżniania „zły e-mail” od „złe hasło”).

- Brak — to fundamentalna historyjka bazowa. S1.2 i S1.7 odwołują się do niej jako „logowanie / konto klienta (auth)”.

## Uwagi implementacyjne

- **Akcje serwerowe** `src/app/(frontend)/[tenant]/account-actions.ts` — `'use server'`: `registerCustomer`, `loginCustomer`, `logoutCustomer` (plus `requestPasswordReset` / `resetPassword` — wiring resetu hasła należy do S1.7). Importuje `next/headers` (`cookies`) **bezpośrednio** zgodnie z pułapką hydratacji use-server / Turbopack.
  - `registerCustomer` tworzy klienta przez Local API (`overrideAccess: true`) ze stemplem `tenant` z bieżącej witryny (`tenantId` — nigdy akceptowany z klienta), następnie loguje i ustawia ciasteczko `payload-token`.
  - `loginCustomer` woła `payload.login`, po czym **wymusza dopasowanie tenanta**: jeśli `tenant` konta ≠ `tenantId` strony, logowanie jest odrzucane (`This account belongs to a different supplier.`), ciasteczko nie jest ustawiane.
  - `logoutCustomer` usuwa ciasteczko `payload-token`.
- **Odczyt sesji** `src/lib/auth.ts` — `getCurrentCustomer(tenantId)` woła `payload.auth({ headers })`, zwraca klienta **tylko** gdy `user.collection === 'customers'` **i** `tenant` konta === `tenantId`. W przeciwnym razie `null` → klient zalogowany u A jest anonimowy u B.
- **Trasa konta** `src/app/(frontend)/[tenant]/konto/page.tsx` — strona konta klienta (rejestracja / login / logout).
- E-mail per-tenant pozostaje TODO (patrz PRD § Poza zakresem): obecnie e-mail jest globalnie unikalny, niedopasowanie tenantów odrzucane przy logowaniu.
