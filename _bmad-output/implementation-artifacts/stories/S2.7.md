# S2.7 — Serwerowa walidacja capacity odporna na wyścig

Status: ☐ TODO · ready-for-dev · EPIC-2 · utworzono 2026-06-21

## Historyjka

Jako **operator** chcę **serwerowej walidacji DOSTĘPNOŚCI slotu (capacity) odpornej na wyścig przy `placeOrder`**, aby **dwoje klientów nie zarezerwowało ostatniego miejsca naraz, a slot pełny znikał z listy** (O4).

## Kryteria akceptacji (z sprint-2.md)

- `placeOrder` waliduje capacity serwerowo i ODPORNIE NA WYŚCIG: atomiczny check-and-reserve / transakcja (Local API) — dwoje klientów na ostatnie miejsce nie przekroczy limitu; przegrany dostaje błąd PL „slot właśnie się zapełnił", zamówienie nie powstaje.
- Wyczerpany slot znika z listy (S2.2); licznik rezerwacji per slot spójny z liczbą **aktywnych** zamówień.
- **Anulowanie (`cancelled`) zwalnia miejsce; reaktywacja `cancelled→new` re-waliduje capacity i odrzuca przy pełnym slocie** (brak nadrezerwacji). Reguła ZATWIERDZONA.
- Rozszerza enforcement z S2.3.

## Uwagi implementacyjne

- Strategia z `SPIKE-S2.md` (d): transakcja Payload Local API; w transakcji `payload.count({ collection: 'orders', where: { and: [{ deliverySlot: id }, { status: { not_in: ['cancelled'] } }] }, req })` → jeśli `>= capacity` rollback + błąd PL, inaczej `create` w tej samej transakcji.
- **Licznik = recount z `orders`** (jedno źródło prawdy, brak driftu); `orders.status` JEST zapytywalny (potwierdzone na live DB w SPIKE-S2 — w przeciwieństwie do `carts.status`).
- **Otwarte do dobrania tutaj (R-S2.7):** mechanizm blokady wiersza — `SERIALIZABLE` z retry **lub** `SELECT … FOR UPDATE` na wierszu slotu jako muteksie przed recountem. Dowód anty-overbooking (dwa równoległe zamówienia na ostatnie miejsce) = test integracyjny tej historii.
