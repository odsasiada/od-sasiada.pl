# Story 2.3: Serwerowa walidacja cutoffu wybranego slotu w `placeOrder`

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

Jako **operator platformy**,
chcę **serwerowej re-walidacji wybranego slotu dostawy w `placeOrder` (cutoff, dzień przeszły, data wyłączona, sensowność)**,
aby **nie dało się złożyć zamówienia na slot zamknięty/przeszły/wyłączony — nawet jeśli klient sfałszuje ciało żądania** (decyzje O2, O7, O8; ryzyko R-S2.2).

## Acceptance Criteria

1. **`placeOrder` RE-waliduje wybrany slot SERWEROWO względem bieżącego czasu** (`now = new Date()` po stronie serwera — NIE z klienta). Walidacja używa **tej samej** czystej `computeAvailableSlots` co ścieżka odczytu (S2.2), więc reguły cutoffu/przeszłości/wyłączonych dat są identyczne (zero driftu). Klient NIE może obejść cutoffu, podstawiając `{ id, date }` w ciele żądania.
2. **Slot nieprawidłowy → błąd PL, zamówienie NIE powstaje.** „Nieprawidłowy" = wybrane wystąpienie `{ id, date }` **nie znajduje się** w zbiorze dostępnych slotów tenanta wyliczonym na `now` (pokrywa: po cutoffie, dzień przeszły/rozpoczęty, data wyłączona O7, slot innego tenanta, slot zdegenerowany, `capacity <= 0`). Komunikat PL np. „Wybrany termin dostawy jest już niedostępny. Odśwież koszyk i wybierz inny termin." `payload.create` zamówienia się NIE wykonuje (return przed create).
3. **O8 — slot wymagany, gdy tenant ma okna.** Gdy tenant ma skonfigurowane sloty (`deliveryEnabled === true`) a klient NIE przekazał slotu → odrzucenie z PL „Wybierz termin dostawy." Gdy tenant NIE ma slotów (`deliveryEnabled === false`) → walidacja slotu **pominięta** (feature off); ewentualny przekazany slot jest ignorowany, checkout działa jak w EPIC-1.
4. **Brak egzekwowania capacity/wyścigu w tej historii.** `reservedCount` nadal = `0` (relacja `deliverySlot` na `orders` nie istnieje — S2.4; recount + atomic check-and-reserve = **S2.7**). Skutek: filtr capacity odrzuca tu wyłącznie `capacity <= 0`. NIE implementować transakcji/blokady wiersza.
5. **Brak zapisu slotu do zamówienia w tej historii.** S2.3 **tylko waliduje**. Snapshot slotu w `orders` (relacja + blok `deliverySlot`, widok w panelu/froncie/mailu) = **S2.4**. `payload.create({ collection: 'orders', ... })` pozostaje bez pola slotu (pole jeszcze nie istnieje).
6. **Walidacja osadzona w istniejącym przepływie `placeOrder`** obok walidacji koszyka/cen (`validateLineItem`), PRZED `payload.create`. Styl błędów PL spójny z istniejącymi (`{ error, ok: false }`, `PlaceOrderResult`). Reużycie wzorca „serwer nie ufa klientowi" z S1.2 (`cart-validation.ts`).
7. **Granica hydratacji nienaruszona (NFR6, R-S2.5)** i zero regresji: `placeOrder` nadal importuje `next/headers` bezpośrednio, typy parametrów inline; `pnpm dev` startuje, checkout działa end-to-end (z oknami i bez okien); `pnpm lint` + `pnpm test` zielone (w tym `orders-isolation`, `order-detail-idor`).

## Dev Notes

### Co to za historia (zakres ścisły)
S2.3 to **wyłącznie serwerowy guard cutoffu/ważności wybranego slotu** w `placeOrder`. Powód istnienia: ścieżka odczytu (S2.2) liczy listę serwerowo, ale klient mógłby podstawić w ciele żądania slot zamknięty/przeszły/sfałszowany — dlatego `placeOrder` musi **re-walidować autorytatywnie** na świeżym `now` (R-S2.2). To dokładnie ta sama filozofia, co re-walidacja cen w S1.2 (klientowi nie ufamy).

**Poza zakresem (świadomie):**
- **S2.7** — recount aktywnych zamówień z `orders` + capacity odporne na wyścig (transakcja, `SELECT … FOR UPDATE`). Tu `reservedCount = 0`.
- **S2.4** — zapis snapshotu slotu w `orders` (relacja `deliverySlot` + widok panel/front/mail). Tu **tylko walidacja**, bez zapisu.
- **S2.2** — UI i przekazanie wyboru (już zrobione/planowane; tu konsumujemy parametr `deliverySlot`).

### Fundament — reużyć, nie pisać od nowa
- **`computeAvailableSlots`** (`src/lib/delivery-slots.ts:150`) — jedyne źródło prawdy dostępności; przetestowane (DST, cutoff, O7, O4, O8). NIE pisać własnej matematyki cutoffu w `placeOrder`.
- **`getAvailableDelivery(tenantId)`** (`src/lib/delivery-slots-read.ts`, z S2.2) — czyta `delivery-slots` + `delivery-date-exceptions` (`overrideAccess: true` + `where { tenant }`), wstrzykuje `new Date()`, zwraca `{ deliveryEnabled, slots }`. **Re-walidacja = ponowne wywołanie tej samej funkcji + sprawdzenie przynależności** wybranego `{ id, date }`. Dzięki temu UI i guard serwerowy nie mogą się rozjechać.
- **Wzór „serwer nie ufa klientowi":** `validateLineItem` (`cart-validation.ts:41`) — typed result `{ ok: true | false, error }`, czytanie autorytatywne z DB. Powtórz ten kształt dla slotu.

### Granica cutoffu i znaczenie membership-check
- Cutoff jest **zamknięty** (`>=`): slot dokładnie o godzinie cutoffu jest już ZAMKNIĘTY (`SPIKE-S2.md` (c), `delivery-slots.ts:206`). Dzięki reużyciu `computeAvailableSlots` dostajesz to za darmo.
- `computeAvailableSlots` zwraca **wyłącznie przyszłe, dostępne** wystąpienia. Jeśli wybrane `{ id, date }` tam jest → slot ważny. Jeśli go nie ma (po cutoffie / przeszły / wyłączona data / obcy tenant / `capacity<=0` / zdegenerowany) → odrzuć. **Jedna kontrola pokrywa wszystkie te przypadki** — nie rozbijaj jej na osobne gałęzie.
- Porównuj `id` przez `String(a) === String(b)` — `AvailableSlot['id']` to `number | string`, a wartość wraca z klienta (mogła zostać przepuszczona przez serializację/`<select>`).

### Gdzie w `placeOrder` (aktualny przepływ — NIE psuć kolejności)
Obecny `placeOrder` (`actions.ts:74`): (1) `resolveCustomer` → (2) tenant + `isActive` → (3) `findCart` + pusty? → (4) `validateLineItem` per linia (równolegle, fold w kolejności) → (5) min order value → (6) `payload.create(orders)` → (7) `payload.update(carts)` czyszczenie. **Wstaw walidację slotu między (5) a (6)** (albo zaraz po (2) dla fail-fast — ale wtedy i tak nie czyścimy koszyka, bo return jest przed create). Kluczowe: **przed `payload.create`**, by odrzucone zamówienie nie powstało i koszyk pozostał nietknięty.

### Granica hydratacji Turbopack (NFR6 / R-S2.5) — KRYTYCZNE
- `actions.ts` jest `'use server'` i **importuje `next/headers` bezpośrednio** (nie przez `@/lib/auth`); typy parametrów trzyma **inline**. To zachować — dodanie `validateChosenSlot`/`getAvailableDelivery` (plain server util, BEZ `next/headers`) jest bezpieczne, bo nie wprowadza pośredniego importu `next/headers` (analogicznie do już importowanego `validateLineItem`).
- NIE nadawać `delivery-slots-read.ts` dyrektywy `'use server'`; NIE importować `next/headers` do modułów czystych (`delivery-slots.ts`) ani do read-utila.
- Złamanie = cicha awaria hydratacji `CartView`. Patrz memory `use-server-turbopack-gotcha`, nagłówek `actions.ts:1`.

### `now` autorytatywne, nie z klienta (sedno R-S2.2)
`getAvailableDelivery` woła `new Date()` na serwerze podczas `placeOrder` → cutoff liczony względem realnego czasu serwera w Europe/Warsaw. Klient nie przekazuje `now` i nie może go sfałszować. Wybrane `{ id, date }` z klienta jest tylko „twierdzeniem", które konfrontujemy ze świeżo wyliczonym zbiorem dostępnych slotów.

### Testowalność bez wstrzykiwania `now`
`placeOrder` używa wewnętrznego `new Date()` (brak parametru `now` — i nie dodawaj go, to niezgodne z resztą akcji). Dlatego testy integracyjne operują na slotach **relatywnych do realnego czasu**: przypadek „prawidłowy" pobiera realne wystąpienie z `getAvailableDelivery`; przypadek „nieprawidłowy" używa **daty z przeszłości** (deterministycznie poza zbiorem dostępnych) lub `capacity: 0`. To stabilne niezależnie od momentu uruchomienia testu.

### Środowisko / workflow
- **Postgres = Docker** (`od-sasiada-pg`, `postgres:17`, port 5432, baza `od_sasiada`) — NIE Homebrew. Jeśli daemon padł: `open -a Docker`, poczekaj aż `docker info` przejdzie, potem `docker start od-sasiada-pg`.
- **Biome** jedyny lint/format: single quotes, bez średników, `organizeImports`/`useSortedKeys` ON — pisz w dowolnej kolejności i odpal `pnpm lint:fix`.
- **Testy:** integracyjne `tests/integration/*.integration.test.ts` (live DB, `fileParallelism: false`, jedna instancja Payload, timeout 60s). Alias `@/` działa w testach.
- **Pieniądze** nadal w groszach + `formatPLN` (dla istniejących komunikatów min. kwoty) — slot nie dotyka kwot.

### Czego NIE robić (anti-patterns)
- ❌ Nie ufać slotowi z klienta — to cała istota tej historii (R-S2.2). `now` z serwera, membership-check ze świeżego zbioru.
- ❌ Nie pisać własnej matematyki cutoffu/przeszłości/wyjątków — reużyj `computeAvailableSlots`/`getAvailableDelivery` (jedno źródło prawdy, brak driftu z S2.2).
- ❌ Nie egzekwować capacity/wyścigu, nie pisać transakcji ani `SELECT … FOR UPDATE` (to S2.7). `reservedCount = 0`.
- ❌ Nie zapisywać slotu do `orders` (pole `deliverySlot` nie istnieje — S2.4).
- ❌ Nie odrzucać przekazanego slotu, gdy `deliveryEnabled === false` (short-circuit `{ ok: true }` — O8 off).
- ❌ Nie nadawać `'use server'` read-utilowi ani nie importować `next/headers` pośrednio do `actions.ts`.
- ❌ Nie czyścić koszyka przed pomyślnym `payload.create` (return przy błędzie slotu następuje wcześniej — pilnuj kolejności).
- ❌ Nie filtrować `carts` po `status` (gotcha — nie dotyczy, ale pamiętaj).
