## 2. Co poszło dobrze

1. **SPIKE-S2 jako fundament (nagłówkowy wniosek — potwierdzony przez Project Lead).** Front-loading wszystkich trudnych decyzji w spike: kształt kolekcji `DeliverySlots`, matematyka cutoffu, strefa Europe/Warsaw + DST, model wyjątków O7, współbieżność capacity O4. Efekt: **zero przeróbek schematu w trakcie sprintu**.
2. **Jedno źródło prawdy `computeAvailableSlots`.** Czysta, deterministyczna, przetestowana funkcja napisana raz w SPIKE; reużywana przez S2.2 (odczyt), S2.3 (walidacja serwerowa), S2.7 (capacity). Cytat z rekordu S2.3: „zero driftu UI↔guard".
3. **Ścisłe scope'owanie historii.** Każda historia z sekcją „Co to za historia / Poza zakresem" + listą anti-patternów + „nie pisz X od nowa". Zapobiegło scope creepowi i duplikacji logiki.
4. **Race-safe capacity (S2.7) — najtrudniejszy problem — rozwiązany czysto.** `pg_advisory_xact_lock(hashtext('slotId|date'))` na sesji transakcji Payloada, per-occurrence, bez pętli retry. Recount z `orders` = jedno źródło prawdy (brak licznika inkrementalnego → brak driftu). Udowodnione testem współbieżności (brama R-S2.7).
5. **Filozofia „serwer nie ufa klientowi" konsekwentna.** S2.3 re-waliduje slot serwerowo na świeżym `now` — ta sama logika co re-walidacja cen w S1.2.
6. **Granica hydratacji Turbopack utrzymana konstrukcyjnie** we wszystkich historiach (plain server util vs `'use server'` vs czysty moduł). Brak cichych awarii hydratacji.
7. **Proaktywny refaktor jakościowy mid-epic.** S2.3 wyciągnął wspólny `tests/setup/delivery-fixtures.ts`, gdy `fallow` wykrył duplikację boilerplate'u testów — usunięte zanim S2.4/S2.7 je zwielokrotniły.
